# 传输抽象层 — 模型无关性

## 1. 设计动机

Hermes Agent 的核心价值之一是**模型无关性（model-agnostic）**——用户可以用 OpenAI、Anthropic、Google Gemini、AWS Bedrock、OpenRouter、本地 Ollama 等 15+ 提供商中的任意一个，代理行为完全一致。

这由 `agent/transports/` 包实现。它不是一个简单的"if-else 选择 API"，而是深度组织了 4 种完全不同的 API 范式。

## 2. 架构：ProviderTransport 抽象基类

### 2.1 接口定义

```python
class ProviderTransport(ABC):
    @abstractmethod
    def convert_messages(messages) -> provider_native_messages
    @abstractmethod
    def convert_tools(tools) -> provider_native_tools
    @abstractmethod
    def build_kwargs(messages, tools, ...) -> api_call_kwargs
    @abstractmethod
    def normalize_response(raw_response) -> NormalizedResponse
```

四个方法构成完整的请求-响应转换管线：

```
OpenAI 格式消息 ──→ convert_messages() ──→ 提供商原生格式
OpenAI 格式工具 ──→ convert_tools()    ──→ 提供商原生格式
                  ──→ build_kwargs()   ──→ 完整 API 调用参数
提供商原始响应 ──→ normalize_response() ──→ 统一的 NormalizedResponse
```

### 2.2 统一响应类型 NormalizedResponse

```python
@dataclass
class NormalizedResponse:
    text: str | None              # 纯文本回复
    tool_calls: List[ToolCall]    # 工具调用列表
    finish_reason: str             # 统一为 OpenAI 风格 stop/tool_calls/length
    usage: dict                    # 统一 token 统计
```

## 3. 四种传输实现

### 3.1 anthropic — Anthropic Messages API

适配 Anthropic 独特的消息结构：

| OpenAI 格式 | Anthropic 格式 |
|---|---|
| `role: "system"` 独立消息 | 顶层 `system` 参数（不在 messages 数组里） |
| `role: "user/assistant"` 交错 | 严格的 `user/assistant` 交替（不允许同角色连续） |
| `function: {name, parameters}` | `input_schema: {type, properties, required}` |
| tool_call 在 assistant 消息的 `tool_calls` 字段 | 独立的 `content_block` 类型 `tool_use` |

**关键处理**：

- **消息合并**：连续的 user 或 assistant 消息需要合并，因为 Anthropic 要求严格交替
- **工具结果格式**：需要包装为 `content: [{type: "tool_result", tool_use_id: ...}]`
- **缓存控制**：`apply_anthropic_cache_control()` 给指定消息添加 `cache_control: {type: "ephemeral"}` 标记

### 3.2 chat_completions — OpenAI 兼容的 Chat Completions API

这是**最通用的传输**，覆盖：

- OpenAI（GPT-4o, GPT-4.1, o-series）
- Google Gemini（通过 OpenAI 兼容端点）
- OpenRouter（200+ 模型）
- Mistral
- Ollama / vLLM 本地模型
- 月之暗面 Moonshot / Kimi
- 任何 OpenAI-compatible 的自建服务

因为 OpenAI Chat Completions 是行业事实标准，这个传输基本是**直通（passthrough）**，转换量最小。但它内部处理了各提供商的细微差异：

| 差异点 | 处理方式 |
|---|---|
| `reasoning_effort`（o-series） | 自动添加到 kwargs |
| Gemma 等不支持 system role 的模型 | 将 system 消息合并到首条 user 消息 |
| 不同提供商的 `max_tokens` 语义差异 | 统一为 `max_completion_tokens`（OpenAI）或 `max_tokens`（其他） |
| 响应 `usage` 字段差异 | `normalize_usage()` 统一格式 |
| 函数名包含特殊字符时的兼容 | sanitize 或 fallback |

### 3.3 bedrock — AWS Bedrock

适配 AWS Bedrock 的独特调用模型：

- **凭证链**：支持 IAM 角色、环境变量、SSO 等多种 AWS 认证
- **流式差异**：Bedrock 的 streaming 格式不同于 OpenAI SSE，需要逐事件解析
- **模型 ID 映射**：`us.anthropic.claude-sonnet-4-20250514-v1:0`
- **thinking 支持**：Claude 的 extended thinking 通过 `additional_model_fields` 传递

### 3.4 codex — GitHub Copilot/Codex

适配 GitHub Copilot 的 Responses API：

- **不同的消息结构**：Responses API 不是 Chat Completions 范式
- **工具调用 ID 确定性生成**：`_deterministic_call_id()` 确保相同输入产生相同 ID
- **不同轮次的语义**：Responses API 的单次调用就能处理多轮工具循环

## 4. 传输选择与切换

### 4.1 自动选择

```python
api_mode = detect_api_mode(base_url, model)
# 检测逻辑：
#   - base_url 包含 "bedrock" → api_mode = "bedrock"
#   - base_url 包含 "codex" → api_mode = "codex"
#   - 使用 Anthropic SDK + base_url 为空 → api_mode = "anthropic_messages"
#   - 默认 → api_mode = "chat_completions"
```

### 4.2 故障转移时的传输切换

当 API 调用失败，AIAgent 可能切换到底层不同的提供商：

```
主凭证池 (OpenAI, api_mode=chat_completions)
  ↓ 失败
备用凭证池 (Anthropic, api_mode=anthropic_messages)
  ↓ 切换
新 transport = transports[new_api_mode]
```

## 5. 提示缓存（Prompt Caching）

`agent/prompt_caching.py` 仅在 Anthropic transport 上生效：

```python
def apply_anthropic_cache_control(messages, model):
    """给系统中可缓存的内容添加 cache_control"""
    # 缓存的候选：
    #   1. 系统提示词（最大收益，每次请求都有）
    #   2. 工具定义（多次调用不变）
    #   3. 历史消息中已经稳定不再变的部分
```

缓存命中的 token 费用通常为基准价格的 10%，显著降低长对话成本。

## 6. 添加新提供商的步骤

1. 在 `agent/transports/` 创建新文件（如 `mistral.py`）
2. 继承 `ProviderTransport`，实现 4 个抽象方法
3. 在 `agent/transports/__init__.py` 注册
4. 在 `agent/model_metadata.py` 添加模型元数据
5. 如果认证方式特殊，在 `agent/credential_sources.py` 添加凭证源

**成本**：约 200-500 行代码即可支持一个新的 LLM 提供商。
