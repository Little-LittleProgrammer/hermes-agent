# 消息网关架构 — 多平台接入

## 1. 设计理念

Hermes Agent 的网关层实现了**一个核心、多端接入**的架构。用户可以从手机、电脑、任何消息平台向代理发消息，代理在远程服务器/容器中执行任务，然后将结果返回给用户。整个流程对代理核心完全透明——它不感知消息来自哪个平台。

## 3. GatewayRunner — 核心编排器

### 3.1 生命周期管理

```python
class GatewayRunner:
    def __init__(self):
        self.adapters = {}          # 平台名 → PlatformAdapter
        self.agent_cache = {}       # session_key → AIAgent (LRU)
        self.session_manager = SessionManager()
        self.stream_consumer = StreamConsumer()
    
    async def start(self):
        # 1. 初始化 Session Manager (SQLite)
        # 2. 根据配置启动启用的平台适配器
        # 3. 启动流消费者
        # 4. 启动投递路由器的定时任务
        # 5. 启动 Agent 缓存清理器
```

### 3.2 Agent 缓存

在 Gateway 中，AIAgent 实例是昂贵的资源（LLM 客户端、工具 schema、记忆提供商），不能每个消息都新建。缓存策略：

```
最大缓存: 128 个实例
驱逐策略: LRU + 空闲 TTL (1小时)
清理器: 每 5 分钟运行一次

_session_expiry_watcher():
    for each cached agent:
        if agent.idle_time > 1h:
            evict(agent)
```

### 3.3 消息处理流程

```
用户发送消息到某个平台
  │
  ├─ 平台 SDK / Webhook 收到原始事件
  │   └─ 平台 Adapter 解析原始 payload
  │       └─ 构造 MessageEvent{text, source, raw_message, media_urls, ...}
  │
  ├─ BasePlatformAdapter.handle_message(event)
  │   ├── 计算 session_key
  │   ├── 处理忙碌会话、排队和中断
  │   └── 后台任务调用 message_handler
  │
  ├─ GatewayRunner._handle_message(event)
  │   ├── 查找/创建 Session
  │   │     └─ session_manager.get_or_create(session_key)
  │   ├── 获取/创建 AIAgent
  │   │     └─ _get_or_create_agent(session_key, user_config)
  │   └── 提交到线程池执行
  │         └─ executor.submit(agent.run_conversation, msg.text)
  │
  ├─ 流式响应
  │   └─ GatewayStreamConsumer.on_delta(delta)
  │       └─ adapter.send / adapter.edit_message 更新平台消息
  │
  └─ adapter.send(chat_id, response)
       └─ 平台格式化、分片、回复线程、媒体发送
```

### 3.4 飞书渠道完整链路

飞书是网关里比较完整的一条平台链路：同一个 `FeishuAdapter` 同时支持 WebSocket 长连接和 Webhook 回调，入站事件会经过鉴权、去重、群聊策略、消息归一化、批处理、会话锁、Agent 执行和飞书消息回复。

#### 3.4.1 启动与连接

```
GatewayRunner.start()
  │
  ├─ GatewayConfig.get_connected_platforms()
  │   └─ 发现 Platform.FEISHU 已启用且配置完整
  │
  ├─ GatewayRunner._create_adapter(Platform.FEISHU, config)
  │   └─ FeishuAdapter(config)
  │
  └─ adapter.connect()
      ├─ 校验 lark-oapi、FEISHU_APP_ID、FEISHU_APP_SECRET
      ├─ acquire_scoped_lock("feishu-app-id", app_id)
      ├─ _build_lark_client(domain)
      ├─ _build_event_handler()
      │   ├─ im.message.receive_v1 → _on_message_event()
      │   ├─ im.message.reaction.* → _on_reaction_event()
      │   ├─ card.action.trigger → _on_card_action_trigger()
      │   └─ drive.notice.comment_add_v1 → _on_drive_comment_event()
      ├─ _hydrate_bot_identity()  # 获取 bot open_id/user_id/name，用于 @mention 判断
      └─ 根据 FEISHU_CONNECTION_MODE 选择：
          ├─ websocket: FeishuWSClient.start() 在独立 executor 线程中运行
          └─ webhook: aiohttp 监听 FEISHU_WEBHOOK_HOST:PORT + FEISHU_WEBHOOK_PATH
```

关键点：

| 阶段 | 说明 |
|---|---|
| **连接模式** | `FEISHU_CONNECTION_MODE=websocket` 使用飞书官方长连接；`webhook` 本地起 aiohttp server 接收事件 |
| **进程锁** | `acquire_scoped_lock("feishu-app-id", app_id)` 防止两个本地 gateway 同时使用同一个飞书应用 |
| **事件注册** | `_build_event_handler()` 把飞书事件类型映射到 adapter 内部 handler |
| **身份缓存** | `_hydrate_bot_identity()` 获取 bot 自身身份，后续群聊 @mention gate 需要它 |

#### 3.4.2 Webhook 安全入口

Webhook 模式先走 `_handle_webhook_request()`，这里不会直接调用 Agent：

```
POST /feishu/webhook
  │
  ├─ IP + path + app_id 维度限流
  ├─ Content-Type / Content-Length / body timeout 校验
  ├─ url_verification 直接返回 challenge
  ├─ verification token 校验
  ├─ x-lark-signature 校验（配置 encrypt_key 时启用）
  ├─ 拒绝 encrypt payload（当前 Hermes webhook mode 不解密）
  └─ 按 header.event_type 分发
      ├─ im.message.receive_v1 → _on_message_event(data)
      ├─ card.action.trigger → _on_card_action_trigger(data)
      └─ 其他已注册事件 → 对应 handler
```

#### 3.4.3 入站消息归一化

无论 WebSocket 还是 Webhook，普通消息最终都会进入同一条路径：

```
Feishu SDK / webhook dispatch
  │
  └─ FeishuAdapter._on_message_event(data)
      ├─ 如果 asyncio loop 暂不可用：放入 pending inbound queue，稍后 replay
      └─ run_coroutine_threadsafe(_handle_message_event_data(data), loop)
          │
          ├─ 取 event.message / event.sender.sender_id
          ├─ message_id 去重：_is_duplicate(message_id)
          ├─ 丢弃 bot 自己发出的消息：_is_self_sent_bot_message(event)
          ├─ 群聊 gate：_should_accept_group_message(message, sender_id, chat_id)
          └─ _process_inbound_message(...)
```

`_process_inbound_message()` 负责把飞书消息变成统一 `MessageEvent`：

```
_process_inbound_message()
  │
  ├─ _extract_message_content(message)
  │   ├─ text/post/interactive/forward/share_chat 归一化为文本
  │   └─ image/file/audio/video 下载或缓存为 media_urls/media_types
  ├─ 文本消息去掉边缘 @bot mention
  ├─ "/" 开头转成 MessageType.COMMAND
  ├─ 回复消息：_fetch_message_text(parent_id / upper_message_id)
  ├─ get_chat_info(chat_id) 获取飞书会话名称和类型
  ├─ _resolve_sender_profile(sender_id) 解析 open_id / union_id / 显示名
  ├─ build_source(...)
  │   ├─ chat_id/chat_name/chat_type
  │   ├─ user_id = open_id
  │   ├─ user_id_alt = union_id（有则用于更稳定的参与者隔离）
  │   └─ thread_id = message.thread_id
  └─ MessageEvent(...)
```

#### 3.4.4 进入通用网关前的飞书专属保护

构造好的 `MessageEvent` 不会马上进 Agent。飞书先做平台侧保护：

```
_dispatch_inbound_event(event)
  │
  ├─ 普通 text → _enqueue_text_event(event)
  │   └─ 同一 session_key 下短时间文本合并，避免飞书客户端连续拆消息触发多轮 Agent
  │
  ├─ 图片/文件/音频/视频 → _enqueue_media_event(event)
  │   └─ 同类附件短时间合并成一个 MessageEvent
  │
  └─ command 或 flush 后事件 → _handle_message_with_guards(event)
      └─ 按 chat_id 加 asyncio.Lock，保证同一飞书会话串行进入通用 pipeline
```

`_handle_message_with_guards()` 最后调用 `BasePlatformAdapter.handle_message(event)`。从这里开始就是所有平台共享的网关行为：计算 `session_key`、处理忙碌会话、排队/中断、后台调用 `GatewayRunner._handle_message(event)`。

#### 3.4.5 Agent 执行与飞书状态反馈

通用 pipeline 在处理开始和结束时会调用 adapter lifecycle hook，飞书用 reaction 表示工作状态：

```
BasePlatformAdapter._process_message_background()
  │
  ├─ adapter.on_processing_start(event)
  │   └─ FeishuAdapter 添加 Typing reaction 到用户原消息
  │
  ├─ GatewayRunner._handle_message(event)
  │   ├─ pre_gateway_dispatch 插件 hook
  │   ├─ 用户授权 / 配对 / slash command
  │   ├─ session 读取与 history 组装
  │   └─ _run_agent(...)
  │       ├─ AIAgent.run_conversation(...)
  │       ├─ tool progress 可通过 adapter.send/edit_message 展示
  │       └─ GatewayStreamConsumer.on_delta(...) 流式编辑飞书消息
  │
  └─ adapter.on_processing_complete(event, outcome)
      ├─ 成功：移除 Typing reaction
      └─ 失败：移除 Typing 后添加 CrossMark reaction
```

#### 3.4.6 飞书回复发送

Agent 的最终回复回到 `FeishuAdapter.send()`：

```
FeishuAdapter.send(chat_id, content, reply_to, metadata)
  │
  ├─ format_message(content)
  ├─ truncate_message(..., MAX_MESSAGE_LENGTH)
  ├─ _build_outbound_payload(chunk)
  │   ├─ Markdown/代码块丰富内容 → post
  │   └─ 普通文本 → text
  ├─ _feishu_send_with_retry(...)
  │   ├─ reply_to 可用：ReplyMessageRequest
  │   └─ reply target 不可用：fallback 到 CreateMessageRequest
  └─ 如果 post payload 被飞书拒绝：降级为 plain text 重发
```

流式输出时，`GatewayStreamConsumer` 会先 `send()` 一个预览消息，再用 `edit_message()` 调用飞书 `message.update` 持续更新；平台不支持或飞书拒绝 `post` 更新时，同样会降级为 plain text。

#### 3.4.7 交互卡片和审批按钮

飞书还把卡片按钮回调接到同一条网关链路：

```
card.action.trigger
  │
  └─ _on_card_action_trigger(data)
      ├─ 审批按钮：_handle_approval_card_action(...)
      │   ├─ 解析 approve_once / approve_session / approve_always / deny
      │   ├─ 回写 resolved card
      │   └─ 通知 tools/approval.py 等待中的审批
      │
      └─ 普通卡片动作：_handle_card_action_event(data)
          ├─ card action token 去重
          ├─ 构造 synthetic_text = "/card <action_tag>"
          ├─ build_source(...)
          └─ MessageEvent(message_type=COMMAND)
              → _handle_message_with_guards(event)
              → BasePlatformAdapter.handle_message(event)
```

这意味着飞书卡片点击最终也会复用同一套 session、授权、slash command 和 Agent pipeline。

## 4. Session Manager (gateway/session.py ~55K)

### 4.1 存储结构 (SQLite WAL + FTS5)

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,        -- 会话 ID
    platform TEXT,               -- 平台名称
    user_id TEXT,                -- 用户 ID
    title TEXT,                  -- 会话标题（AI 生成）
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    metadata JSON
);

CREATE VIRTUAL TABLE session_messages_fts USING fts5(
    content,                     -- 消息内容全文索引
    session_id UNINDEXED,
    tokenize='porter unicode61'
);
```

### 4.2 关键功能

| 功能 | 说明 |
|---|---|
| **会话隔离** | 每个用户-平台组合可以有多个会话，互不干扰 |
| **FTS5 全文搜索** | Agent 可以搜索自己的历史对话 |
| **标题生成** | `agent/title_generator.py` 用辅助模型自动生成会话标题 |
| **消息重放** | 切换模型/重置上下文时可以重放历史消息 |
| **会话迁移** | 支持将会话导出/导入（`claw migrate` 命令） |

## 5. GatewayStreamConsumer (gateway/stream_consumer.py)

负责将 AIAgent 的流式输出分发到正确的目标：

```python
class GatewayStreamConsumer:
    def on_delta(self, text):
        """同步回调：接收 agent worker thread 发来的 delta chunk"""
        self._queue.put(text)
    
    async def run(self):
        """异步任务：缓冲、限速，并通过 adapter.send/edit_message 更新平台消息"""
        ...

    def finish(self):
        """通知流式输出完成，run() 做最后一次刷新"""
        self._queue.put(_DONE)
```

**批量刷新策略**：不是每个 token 立即发送，而是基于时间（~200ms）或换行符触发刷新，减少 API 调用次数。

## 6. 平台适配器

### 6.1 抽象基类

```python
class BasePlatformAdapter:
    platform_name: str                    # "telegram", "discord" 等
    session_manager: SessionManager
    
    async def connect(self)                  # 启动连接
    async def disconnect(self)               # 停止连接
```

### 6.3 添加新平台的步骤

优先使用插件路径，尤其是社区或第三方平台：

1. 在 `~/.hermes/plugins/<name>/` 或仓库的 `plugins/platforms/<name>/` 创建插件目录
2. 添加 `plugin.yaml`，平台插件设置 `kind: platform`
3. 实现继承 `BasePlatformAdapter` 的 adapter
4. 在 `register(ctx)` 中调用 `ctx.register_platform(...)`

平台插件注册后，网关会通过 `gateway/platform_registry.py` 发现 adapter，并复用同一套配置解析、授权、cron 投递、`send_message` 路由、状态展示、setup 向导和平台提示词机制。仓库内的 `plugins/platforms/irc/` 是当前参考实现。

内置平台才需要改核心文件。此时参考 `gateway/platforms/ADDING_A_PLATFORM.md`，通常要覆盖 `gateway/config.py` 的 `Platform`/环境变量读取、`gateway/run.py` 的 adapter factory、授权映射、提示词、toolset、cron 和 `send_message` 等集成点。

## 8. 镜像模式 (gateway/mirror.py ~6K)

支持**跨平台会话镜像** — 同一个 AIAgent 的对话可以在多个平台之间同步：

```
用户 A 在 Telegram 上开始对话
  → 用户 B 在 Discord 上加入同一会话
  → 两人看到相同的 Agent 响应
  → 两人的消息都发送给同一个 AIAgent
```

## 9. 配对系统 (gateway/pairing.py ~12K)

安全的设备配对流程：

```
1. 用户运行 `hermes pair` 获取配对码
2. 在目标平台上发送配对指令
3. Gateway 验证配对码 → 建立绑定
4. 会话持久化，设备可安全断开
```
