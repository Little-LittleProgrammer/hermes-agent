# 近期架构增量与新能力

这篇文档是对 `docs/developer-guide/00-12` 的增量补充，重点覆盖近一段时间已经进入主干、但旧章节还没有系统展开的能力。它不替代原有章节，而是回答两个问题：

1. 现在这套系统比旧文档多了什么？
2. 新能力分别落在哪些源码入口上？

---

## 1. 先看结论：近期最重要的四类变化

### 1.1 Profiles：从“一个 Hermes 实例”变成“同机多身份、多隔离实例”

Hermes 现在不是只有一个 `~/.hermes`。它支持 **profile** 模式：每个 profile 都有独立的 `config.yaml`、`.env`、memory、sessions、skills、logs、cron 和 gateway 状态。

典型用法：

```bash
hermes profile create coder
hermes profile create writer --clone
hermes -p coder chat
hermes profile use coder
```

核心落点：

- `hermes_cli/profiles.py`
- `hermes_constants.py`
- `hermes_cli/main.py`

关键设计点：

- 默认 profile 仍然是 `~/.hermes`，兼容旧用户。
- 命名 profile 默认位于 `~/.hermes/profiles/<name>/`。
- 代码里凡是读写 Hermes 状态的地方，都应该走 `get_hermes_home()`，不能硬编码 `Path.home() / ".hermes"`。
- profile 还额外隔离了 subprocess HOME，避免 `git`、`ssh`、`gh`、`npm` 之类工具在不同 profile 之间串状态。

这意味着今天的 Hermes 更像“一个 agent runtime + 多个长期身份”，而不是单实例聊天工具。

### 1.2 TUI 与 Dashboard：主聊天界面已经从经典 CLI 扩展成“同一套 TUI，多入口复用”

旧文档里对 CLI 的描述仍以 `prompt_toolkit` 交互为主，但当前主线已经同时存在两套前端：

- 经典 CLI：`cli.py`
- 新 TUI：`ui-tui/` + `tui_gateway/`

其中最重要的变化不是“换了个 UI”，而是 **Dashboard `/chat` 页并没有重写聊天逻辑**，而是直接嵌入真实的 `hermes --tui`。

核心链路：

```text
Browser /chat
  -> xterm.js
  -> /api/pty WebSocket
  -> hermes_cli/pty_bridge.py
  -> hermes --tui child process
  -> tui_gateway/*
  -> AIAgent
```

关键源码：

- `ui-tui/src/*`
- `tui_gateway/server.py`
- `tui_gateway/entry.py`
- `hermes_cli/pty_bridge.py`
- `hermes_cli/web_server.py`

关键设计点：

- Dashboard 聊天页复用真实 TUI，因此 TUI 上的新能力会自动进入 Web 聊天页。
- `tui_gateway` 通过 JSON-RPC 向 Ink 前端发事件，例如 `gateway.ready`、`skin.changed`、tool progress、approval prompt。
- Dashboard 侧不是“第二套 Agent UI”，而是 PTY 承载层 + 周边辅助面板。
- 皮肤系统已经是跨 CLI/TUI 联动的；TUI 启动后会接收 `gateway.ready` 下发的 skin 数据。

如果你要改聊天体验，优先看 `ui-tui/` 和 `tui_gateway/`，而不是重新在 React 里发明一套 transcript/composer。

### 1.3 记忆系统已经拆成“双层内置记忆 + 单选外部 provider + 单选 context engine”

旧文档把记忆系统讲成了“MemoryManager + ContextCompressor”，这个方向没错，但现在的真实结构更明确了：

1. **内置记忆**  
   `tools/memory_tool.py` 管理 `MEMORY.md` / `USER.md` 两个受限大小的文件型长期记忆。

2. **外部记忆 provider（最多一个）**  
   通过 `plugins/memory/<name>/` 加载，例如 Honcho、Mem0、Supermemory、RetainDB、OpenViking、Hindsight、Holographic、ByteRover。

3. **上下文引擎（最多一个）**  
   通过 `plugins/context_engine/<name>/` 加载；默认仍是内置 compressor，但框架已经允许替换。

4. **会话搜索**  
   独立于 memory provider，基于 `hermes_state.py` 的 SQLite FTS5 / trigram FTS。

核心源码：

- `agent/memory_manager.py`
- `agent/memory_provider.py`
- `plugins/memory/__init__.py`
- `agent/context_engine.py`
- `plugins/context_engine/__init__.py`
- `tools/memory_tool.py`
- `hermes_state.py`

### 1.4 Kanban：系统已经有了“跨 profile 的持久化多代理协作板”

这是近期最重要的新能力之一。Hermes 现在不只支持一次性的 `delegate_task`，还支持一个 **持久化、SQLite 驱动、跨 profile 的 Kanban 协作板**。

核心组件：

- `hermes_cli/kanban.py`：CLI 与 slash 命令入口
- `hermes_cli/kanban_db.py`：SQLite 状态机与事务核心
- `tools/kanban_tools.py`：worker 进程内可见的结构化 kanban 工具
- `plugins/kanban/dashboard/`：Dashboard 板视图
- `gateway/run.py`：默认内嵌 dispatcher / notifier
- `docs/hermes-kanban-v1-spec.pdf`：完整规格

它解决的问题不是“子代理能不能帮我做事”，而是：

- 多个长期 profile 怎样协作？
- 任务怎样跨重启持续存在？
- 人怎样介入 blocked / review / retry 流程？
- 编排者怎样只拆解任务，不自己下场执行？

## 2. Profiles：多实例隔离已经是基础设施能力

### 2.1 Profile 不只是配置切换，而是整套状态隔离

`hermes_cli/profiles.py` 的实现说明，profile 的隔离范围包括：

- `config.yaml`
- `.env`
- `memories/`
- `sessions/` 与会话状态
- `skills/`
- `skins/`
- `logs/`
- `plans/`
- `cron/`
- `workspace/`
- `home/`（供 subprocess 使用）

这比“多套模型配置”更重。它的真实定位是：**同一台机器上运行多个长期身份不同、记忆不同、凭证不同的 Hermes 实例**。

### 2.2 为什么很多代码必须改成 profile-safe

以前直接写：

```python
Path.home() / ".hermes"
```

现在会出错，因为：

- 默认 profile 确实还是这个路径；
- 但 `coder`、`writer` 之类 profile 的状态根目录并不在这里。

因此：

- 代码路径用 `get_hermes_home()`
- 用户展示文案用 `display_hermes_home()`

如果是 profile 列表/根目录管理，又要回到默认 root，而不是当前 active profile 的 home。

### 2.3 对测试和插件的影响

这件事直接影响：

- 测试夹具要同时 mock `Path.home()` 和 `HERMES_HOME`
- 插件不能偷偷写 `~/.hermes/...`
- gateway 这类长生命周期组件要能在 profile 间正确拿锁、分会话、分日志

如果某个新功能需要落盘，又没有经过 profile-safe 设计，后面大概率会出串号问题。

## 3. TUI、PTY Bridge 与 Dashboard：现在的聊天面已经是三层结构

### 3.1 经典 CLI 仍在，但它不再代表全部交互面

`cli.py` 仍然很重要，负责：

- slash commands
- classic prompt_toolkit 交互
- skin engine
- 某些 setup / config / status 路径

但当前用户体验不再只由它决定。对“聊天面”来说，更重要的是：

- `ui-tui/`：Ink 前端
- `tui_gateway/`：Python JSON-RPC 后端
- `hermes_cli/pty_bridge.py`：Dashboard 的 PTY 适配层

### 3.2 TUI 与 Dashboard 的关系

最容易误解的一点是：Dashboard 不是单独重写了一套聊天 UI。

真实关系是：

1. 浏览器打开 `/chat`
2. 前端用 xterm.js 连 `/api/pty`
3. 服务端通过 `PtyBridge` 启一个真实的 `hermes --tui`
4. 这个 TUI 子进程内部再通过 `tui_gateway` 驱动 AIAgent

所以：

- 聊天主体验仍归 TUI
- Dashboard 只是在浏览器里承载这套终端会话
- 附加的 React 侧栏、模型选择、状态面板可以存在，但不应该复制 transcript/composer 主逻辑

### 3.3 为什么 `PtyBridge` 很关键

`hermes_cli/pty_bridge.py` 代表当前 Web Chat 的设计边界：

- 仅支持 POSIX PTY；Windows 需要 WSL
- 服务端不依赖 Node
- 直接传递 ANSI 字节流
- 浏览器看到的是一个真实终端，不是伪装出来的聊天室

这个选择带来的好处是：

- 新增 TUI 功能时，CLI TUI 和 Dashboard 聊天页自动一致
- slash commands、审批交互、tool rows、skin、prompt 等都天然复用

代价是：Web 聊天页的很多行为要顺着 PTY/TUI 去理解，而不是从普通 Web app 视角重写。

### 3.4 TUI 的事件面

`tui_gateway/server.py` 暴露的是一个事件化协议层，核心事件包括：

- `gateway.ready`
- `skin.changed`
- `message.delta`
- `message.complete`
- `tool.start / progress / complete`
- approval / clarify / secret / sudo prompt

因此今天如果你要扩展 TUI：

- 视觉渲染多半改 `ui-tui/src/*`
- Python 侧交互协议改 `tui_gateway/server.py`
- 想让 Dashboard 自动获得新行为，就别绕开这两层

## 4. 记忆、会话搜索与上下文引擎：今天的真实分层

### 4.1 内置记忆已经收敛成两个文件

`tools/memory_tool.py` 管理两个文件：

- `memories/MEMORY.md`
- `memories/USER.md`

特点：

- 有字符上限，不是无限笔记库
- session start 时作为 frozen snapshot 注入 system prompt
- mid-session 写入会立即落盘，但不会修改当前 session 的系统提示词快照
- 带内容安全扫描，防 prompt injection / secret exfiltration

这套设计是为了在“长期记住关键信息”和“不要打坏 prompt caching”之间折中。

### 4.2 外部 memory provider 现在是正式的一等扩展面

`agent/memory_provider.py` 已经把 provider 生命周期抽象完整了：

- `initialize()`
- `system_prompt_block()`
- `prefetch()`
- `queue_prefetch()`
- `sync_turn()`
- `get_tool_schemas()`
- `handle_tool_call()`
- `shutdown()`

而 `agent/memory_manager.py` 负责：

- 内置 provider 永远启用
- 最多再挂一个外部 provider
- 每轮对话前做 recall / prefetch
- 每轮对话后做 sync
- 将 provider 返回内容包进 `<memory-context>`，并提供 scrubber 防泄漏到流式输出

这说明 memory provider 已经不是“小插件”，而是 agent 主循环上的标准挂点。

### 4.3 Context engine 已经从 compressor 抽象成单选引擎

`agent/context_engine.py` 定义了新的边界：

- 何时压缩
- 如何压缩
- 怎样跟踪 token 使用
- 是否提供额外工具

而 `plugins/context_engine/__init__.py` 说明：

- context engine 和 general plugin 是两条独立发现链
- 同时只能启一个
- 默认是 `compressor`

所以今天“上下文压缩”不再只是一个内部模块，而是一个可以被替换的策略层。

### 4.4 Session Search 不是向量库，而是 SQLite FTS5 / trigram recall

`hermes_state.py` 现在承担：

- session/message 持久化
- `messages_fts`
- `messages_fts_trigram`
- 统一的 `search_messages()`

这件事要和 memory provider 区分开：

- `memory`：始终在上下文里的关键长期事实
- `session_search`：按需召回历史对话
- `external provider`：语义长期记忆、用户建模、知识图谱、rerank 等

对 CJK 场景特别重要的一点是：已经有 trigram FTS 路径，而不只是默认 tokenizer。

## 5. Kanban：多代理协作已经有独立状态机，而不是只靠 `delegate_task`

### 5.1 为什么 Kanban 是新增的一层，而不是 `delegate_task` 别名

`delegate_task` 的形状更像一次 RPC：

- 父任务调用子任务
- 等子任务返回
- 上下文内联回父对话

Kanban 不是这个模型。`hermes_cli/kanban_db.py` 表达的是一个**持久化状态机**：

- task
- task_links
- task_comments
- task_events
- task_runs

状态集至少包括：

- `triage`
- `todo`
- `ready`
- `running`
- `blocked`
- `done`
- `archived`

这意味着它处理的是“工作流编排”，而不是单轮子代理调用。

### 5.2 为什么它天然适合多 profile 协作

Kanban board 存在 `kanban.db`，并且刻意是 profile-agnostic 的共享协调面。

好处：

- `researcher` 可以建任务给 `writer`
- `reviewer` 可以重新接手 `backend-eng` 的产出
- 人可以在 `blocked` 时插入 comment 或解除阻塞
- 任务 crash / reclaim / retry 有独立 run 历史

这跟“父代理上下文里 fork 一个匿名子代理”是两种完全不同的协作语义。

### 5.3 Gateway 已经内嵌 dispatcher / notifier

这点很关键。Kanban 不是一个只存在于 CLI 的本地玩具。

`gateway/run.py` 里已经有后台 watcher：

- notifier watcher
- dispatcher watcher

也就是说，只要 gateway 跑着，Kanban 就可以自动：

- 轮询 ready 任务
- claim
- spawn 对应 profile worker
- 回收超时或崩掉的 worker
- 推送完成/阻塞/失败通知

这使 Kanban 成为 gateway 时代的“长期工作编排层”。

### 5.4 Worker 不是 shelling out，而是有专用 kanban tools

`tools/kanban_tools.py` 的设计非常成熟，原因写得很明确：

- worker 可能运行在 Docker / Modal / SSH / Singularity 后端
- 那里未必有本地 `hermes` 命令
- 也未必挂载了宿主机的 `kanban.db`

所以 worker 通过结构化工具与 Kanban 内核交互：

- `kanban_show`
- `kanban_complete`
- `kanban_block`
- `kanban_heartbeat`
- `kanban_comment`
- `kanban_create`
- `kanban_link`

并且这些工具只有在 `HERMES_KANBAN_TASK` 存在时才进 schema，普通聊天会话完全看不到。

这说明 Kanban 已经不只是“CLI 功能”，而是主 agent 运行时的一部分。

### 5.5 Dashboard 也已经有 Kanban 板插件

仓库里已有：

- `plugins/kanban/dashboard/`
- `website/docs/user-guide/features/kanban.md`
- `website/docs/user-guide/features/kanban-tutorial.md`
- `docs/hermes-kanban-v1-spec.pdf`

所以当前 Kanban 有三套正式入口：

- CLI：`hermes kanban ...`
- slash command：`/kanban ...`
- Dashboard board

它已经是完整产品面，而不是实验性脚本。

## 7. 阅读旧文档时要主动修正的几处认知

### 7.1 “CLI = 主要交互面”

现在不够准确。更准确的说法是：

- `cli.py` 仍是重要入口
- 但聊天体验已经分成 classic CLI 与 TUI 两条主线
- Dashboard `/chat` 复用的是真实 TUI，而不是另一套 Web chat

### 7.2 “记忆系统 = 文件记忆 + 压缩器”

也不够准确。现在至少应该理解为：

- 内置文件记忆
- 单选外部 memory provider
- 单选 context engine
- 独立 session_search

### 7.3 “多代理协作 = delegate_task”

现在也不够准确。当前已经有两种机制：

- 短生命周期、父子式：`delegate_task`
- 长生命周期、持久化、跨 profile：Kanban

### 7.4 “所有插件都走同一套发现机制”

不对。现在至少有三条面：

- general plugins
- memory providers
- context engines

它们的发现路径和选择约束不完全相同。

---

## 8. 一句话总结

如果说旧版 Hermes 更像“一个会调工具的单实例 agent”，那么现在的 Hermes 已经明显演进成：

**一个支持多 profile 长期身份、可切换记忆后端、可替换上下文引擎、拥有真实 TUI/Web 双前端，并且带持久化多代理协作板的 agent runtime。**

这也是为什么后续读代码时，不能再只盯着 `run_agent.py + cli.py + tools/` 三件套；`profiles`、`tui_gateway`、`plugins/memory`、`plugins/context_engine`、`kanban_*` 已经进入主干架构层。
