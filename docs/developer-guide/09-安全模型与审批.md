# 安全模型与审批

## 1. 多层安全架构

Hermes Agent 直面一个核心矛盾：**Agent 需要有强大的能力（执行命令、访问文件、浏览网页），但必须防止滥用**。安全模型采用纵深防御（defense in depth）策略，每一层独立工作且相互补充。

```
┌───────────────────────────────────────┐
│ 第 1 层：提示注入检测                  │  ← 上下文文件扫描
├───────────────────────────────────────┤
│ 第 2 层：命令审批系统                  │  ← 危险命令检测 + 人工/LLM审批
├───────────────────────────────────────┤
│ 第 3 层：文件系统安全                  │  ← 路径限制 + 敏感文件保护
├───────────────────────────────────────┤
│ 第 4 层：网络安全                      │  ← URL 安全检查
├───────────────────────────────────────┤
│ 第 5 层：容器隔离                      │  ← Docker/Modal/Daytona 沙箱
├───────────────────────────────────────┤
│ 第 6 层：TIRITH 集成（可选）           │  ← 企业级安全合规框架
└───────────────────────────────────────┘
```

## 2. 提示注入检测

### 2.1 上下文文件扫描

`agent/prompt_builder.py` 中的 `_scan_context_content()` 在注入任何上下文文件（HERMES.md, SOUL.md）前进行扫描：

| 检测模式 | 风险类型 |
|---|---|
| `ignore previous/all instructions` | 提示注入 |
| `system prompt override` | 系统提示词覆盖 |
| `disregard your instructions/rules` | 规则绕过 |
| HTML 注释注入（`<!-- ... -->`） | 隐藏注入 |
| CSS hidden div | 视觉隐藏注入 |
| translate + execute | 翻译执行攻击 |
| `curl ... ${KEY/TOKEN/SECRET}` | 凭据窃取 |
| `cat .env/credentials` | 敏感文件读取 |

### 2.2 不可见字符检测

```python
_CONTEXT_INVISIBLE_CHARS = {
    '​',  # Zero-width space
    '‌',  # Zero-width non-joiner
    '‍',  # Zero-width joiner
    '⁠',  # Word joiner
    '﻿',  # BOM
    '‪'-'‮',  # Bidirectional text control
}
```

如果检测到任何威胁，上下文文件内容**完全不被加载**，替换为 `[BLOCKED: ...]` 消息。

## 3. 命令审批系统 (tools/approval.py ~52K)

### 3.1 危险命令检测

```python
DANGEROUS_PATTERNS = [
    # 文件系统危险操作
    (r'rm\s+-rf\s+/', 'critical'),
    (r'>\s*/dev/sd[a-z]', 'critical'),
    
    # 系统危险操作
    (r'shutdown|reboot|init\s+[06]', 'critical'),
    (r'mkfs\.|dd\s+if=', 'critical'),
    
    # 网络危险操作
    (r'iptables\s+-F|ufw\s+disable', 'critical'),
    
    # 权限相关
    (r'chmod\s+777', 'critical'),
    (r'chown\s+-R\s+', 'critical'),
    
    # 高危模式
    (r'curl.*\|.*(ba)?sh', 'critical'),
    (r'eval\s+', 'high'),
    
    # 等等... (50+ 模式)
]
```

每个命令被分为三个风险等级：
- **critical**：必须由用户手动审批
- **high**：LLM 自动评估后决定
- **medium**：仅在特定上下文需要审批

### 3.2 审批会话状态

```python
class ApprovalSession:
    """每个会话独立的审批状态"""
    approve_all_until: float | None     # 期限前自动批准
    denied_patterns: Set[str]            # 永久拒绝的模式
    allowed_patterns: Set[str]           # 永久允许的模式
    smart_approval_enabled: bool         # 是否启用 LLM 智能审批
```

**持久化**：允许/拒绝列表可以写入 `~/.hermes/config.yaml`，跨会话持久化。

### 3.3 LLM 智能审批

```python
def smart_approval_check(command, context):
    """使用辅助 LLM 判断命令是否真正危险"""
    prompt = f"""
    You are a security reviewer. Evaluate this command in context:
    
    Command: {command}
    Context: {context}
    
    Is this command dangerous? Consider:
    1. Does it modify system files?
    2. Does it access sensitive data?
    3. Does it have network security implications?
    
    Respond with: SAFE, CAUTION, or DANGEROUS
    """
    return call_auxiliary_llm(prompt)
```

### 3.4 审批流程

```
Agent 要执行命令
  │
  ├─ 检查命令是否匹配 DANGEROUS_PATTERNS
  │   └─ 不匹配 → 直接执行
  │
  ├─ 检查会话审批状态
  │   ├── 在 allowed_patterns 中 → 直接执行
  │   └── 在 denied_patterns 中 → 拒绝执行
  │
  ├─ 判断风险等级
  │   ├── critical → 必须人工审批
  │   ├── high → LLM 智能审批评估
  │   └── medium → 根据上下文决定
  │
  └─ 审批交互
      ├── CLI 模式 → 终端 prompt（y/n/always/never）
      └── Gateway 模式 → 消息平台上的 inline button 审批
```

### 3.5 插件钩子

审批系统对外暴露了两个插件生命周期钩子：

| 钩子 | 时机 |
|---|---|
| `pre_approval_request` | 审批请求发到用户之前 |
| `post_approval_response` | 用户做出审批决定之后 |

## 4. 文件系统安全

### 4.1 路径安全检查 (tools/path_security.py)

```python
def validate_path(path, allowed_roots=None):
    """确保文件路径在允许的根目录范围内"""
    # 防止路径遍历攻击
    real_path = os.path.realpath(path)
    for root in allowed_roots:
        if real_path.startswith(root):
            return True
    return False
```

### 4.2 敏感文件保护 (tools/file_safety.py)

自动检测和阻止对敏感文件的访问：
- `.env`, `.git-credentials`, `.netrc`
- SSH 私钥 `~/.ssh/id_*`
- 云服务凭据文件（`~/.aws/credentials`, `~/.config/gcloud/`）
- macOS Keychain 文件

### 4.3 二进制文件检测 (tools/binary_extensions.py)

防止 Agent 无意中读取/修改二进制文件导致损坏。

### 5.2 凭据池隔离 (agent/credential_pool.py ~61K)

API 凭据的管理：

```python
class CredentialPool:
    """多凭证池 — 故障隔离 + 凭据轮换"""
    pools: Dict[str, List[Credential]]  # 提供商 → 凭证列表
    active: Dict[str, Credential]        # 当前活跃凭证
    
    def rotate(self, provider, reason):
        """切换到下一个凭证（故障转移）"""
    
    def get_client(self, provider):
        """获取当前活跃凭证的客户端实例"""
```

- 凭证永远不会被 Agent 看到（LLM 无法读取）
- 故障时自动切换到备用凭证
- 支持环境变量、配置文件、外部密钥管理等多种凭证来源

## 6. TIRITH 安全框架 (tools/tirith_security.py ~26K)

TIRITH 是可选的企业级安全合规框架集成，提供：

- **审计跟踪**：所有 Agent 操作的完整记录
- **合规策略**：预定义的行业合规策略模板
- **风险评分**：实时评估 Agent 行为的风险等级
- **事件响应**：检测到威胁时的自动响应

## 7. 网站策略 (tools/website_policy.py)

自动检测和遵守网站的 `robots.txt` 和服务条款：
- 检查目标网站是否允许自动化访问
- 遵守 `X-Robots-Tag` HTTP 头
- 在需要时触发人工确认
